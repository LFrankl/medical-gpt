# 借助 LLM 生成高质量训练数据操作手册

这份文档专门解决一个问题：

怎么借助 LLM，稳定地生成可以用于这个项目的高质量数据。

重点不是“让 LLM 多产一点”，而是“让 LLM 在你的规则下，产出可控、可筛、可复用的数据”。

适用范围：

- SFT 数据生成
- 偏好数据生成
- 数据审核与清洗
- 评测集扩写

## 1. 基本原则

不要把 LLM 当成“自动数据金矿”，而要把它当成“高效率的数据工人”。

你负责：

- 定任务
- 定标准
- 定格式
- 定审核规则

LLM 负责：

- 扩写
- 改写
- 初步打标
- 生成候选答案
- 生成难例
- 审核初稿

最稳的方式不是单轮生成直接入库，而是：

1. 人工写少量高质量种子
2. LLM 扩写
3. 第二轮 LLM 审核
4. 人工抽检
5. 小规模训练验证
6. 根据 bad cases 回补数据

## 2. 先拆任务，不要先堆数据

医疗场景下，建议至少拆成这些任务桶：

- `medical_qa`
- `triage`
- `follow_up`
- `drug_safety`
- `report_explanation`
- `emergency_red_flags`
- `uncertainty_and_refusal`

同一个任务桶里，再分别准备：

- `sft`
- `preference`
- `eval`

不要一开始就做“泛医疗问答大杂烩”。

## 3. 数据生产总流程

建议固定成下面这条流水线：

1. 定义任务桶和标签
2. 写 50 到 200 条人工种子样本
3. 用 LLM 扩写成候选池
4. 用审核 LLM 做质量筛查
5. 去重、去模板化、分桶平衡
6. 人工抽检 10% 到 20%
7. 转成训练格式
8. 小样本训练
9. 收集 bad cases
10. 再用 LLM 回补针对性数据

真正高质量的数据，大量来自第 9 步，不是第一轮凭空生成。

## 4. 你应该让 LLM 做什么

最适合让 LLM 做的事：

- 生成问题变体
- 生成多轮追问链
- 生成高质量答案初稿
- 生成优劣回答对
- 生成不同风险等级的病例表达
- 做结构化标注
- 按规则审核和筛掉坏样本

不适合直接放权的内容：

- 高风险医学决策
- 用药剂量建议
- 严重疾病鉴别诊断
- 未经审核的处方级内容

## 5. SFT 数据生成模板

### 5.1 目标

SFT 数据的作用是教模型“如何回答”。  
重点是：

- 结构稳定
- 风格一致
- 谨慎克制
- 会提示边界
- 高风险时提醒就医

### 5.2 推荐输入材料

先准备人工种子样本，每条至少包含：

- 用户问题
- 标准答案
- 任务桶
- 风险等级

例如：

```json
{
  "task_type": "triage",
  "risk_level": "medium",
  "question": "孩子发烧到 38.8 度怎么办？",
  "answer": "38.8 度属于发热，处理方式要结合年龄、精神状态以及是否伴有抽搐、呼吸困难等情况综合判断。如果精神状态尚可，可先补液、休息，并在医生指导下使用退热药。若高热持续不退、精神差、抽搐或呼吸异常，应尽快就医。"
}
```

### 5.3 SFT 扩写 Prompt 模板

下面这段可以直接用，按你的任务替换种子样本内容：

```text
你现在是医学训练数据生成助手。

任务：
基于给定的种子样本，生成 5 条新的高质量 SFT 训练样本。

目标：
这些样本将用于训练一个医学问答模型，因此回答必须谨慎、切题、结构清晰。

约束：
1. 不要直接照抄种子样本。
2. 允许改写病人描述、年龄、场景和表达方式，但必须保持医学上合理。
3. 回答不要抢诊断，不要给出过度确定的结论。
4. 回答应尽量包含：
   - 可能性或判断边界
   - 需要进一步确认的信息
   - 一般性建议
   - 危险信号和就医提醒
5. 不要给出处方剂量或高风险具体治疗方案。
6. 不要生成明显错误或不安全建议。
7. 输出必须为 JSON 数组。
8. 每条数据字段必须为：
   - task_type
   - risk_level
   - conversations
9. conversations 格式必须兼容 ShareGPT：
   [{"from":"human","value":"..."},{"from":"gpt","value":"..."}]

种子样本：
{{seed_samples}}
```

### 5.4 多轮对话扩写 Prompt 模板

如果你要专门生成医患多轮对话，用这个版本更稳：

```text
你现在是医学多轮对话训练数据生成助手。

任务：
围绕给定主题，生成 3 条多轮医患对话数据，用于 SFT 训练。

要求：
1. 每条对话为 2 到 4 轮。
2. 第一轮由患者描述问题。
3. 医生回答时要体现追问意识，不要一上来下结论。
4. 后续轮次应根据新信息调整建议。
5. 最后一轮回答必须包含边界提醒或就医建议。
6. 输出为 JSON 数组，每条仅包含：
   - task_type
   - risk_level
   - conversations
7. conversations 使用：
   {"from":"human","value":"..."}
   {"from":"gpt","value":"..."}

主题：
{{topic}}
```

### 5.5 SFT 审核 Prompt 模板

生成后的 SFT 候选样本不要直接入库，先过审核：

```text
你现在是医学训练数据审核员。

请审核下面的 SFT 样本，并按以下标准逐条判断：
1. 是否有明显医学错误
2. 是否存在不安全建议
3. 是否过度自信地下诊断
4. 是否缺少必要的风险提醒
5. 是否答非所问
6. 是否过于模板化或重复

输出要求：
对每条样本输出：
- id
- verdict: PASS / FAIL / REVISE
- issues: 问题列表
- revised_sample: 若需要修改则给出修订后的完整样本，否则置空

待审核样本：
{{candidate_samples}}
```

## 6. Preference 数据生成模板

### 6.1 目标

偏好数据不是为了“再写一次标准答案”，而是让模型学会区分：

- 哪个回答更准确
- 哪个更安全
- 哪个更切题
- 哪个更像你想要的助手

### 6.2 最好的来源不是纯生成，而是“真实候选回答比较”

推荐来源：

1. `base model` 输出
2. `SFT model` 输出
3. 同一模型不同温度输出
4. 人工写一个强答，再保留一个弱答

最好的偏好对，往往来自真实模型输出，而不是 LLM 人工编造一个“特别蠢”的 rejected answer。

### 6.3 偏好判断 Prompt 模板

```text
你现在是医学回答质量评审员。

请比较两个回答 A 和 B，按以下优先级判断谁更好：
1. 准确性
2. 安全性
3. 切题程度
4. 信息完整性
5. 表达清晰度

规则：
1. 不允许因为回答更长就默认更好。
2. 如果一个回答更礼貌但医学内容更差，不能选它。
3. 如果一个回答对危险情况缺少提醒，优先判为较差。
4. 如果两者质量接近且无法稳定区分，输出 TIE。

输出格式：
{
  "winner": "A" | "B" | "TIE",
  "reason": "简短说明"
}

问题：
{{question}}

回答A：
{{answer_a}}

回答B：
{{answer_b}}
```

### 6.4 直接生成偏好对 Prompt 模板

如果你暂时没有多模型候选输出，也可以让 LLM 先生成一强一弱的对比样本，但这一步只适合早期冷启动。

```text
你现在是医学偏好数据生成助手。

请围绕给定问题生成两个回答：
- 一个为高质量回答，作为 chosen
- 一个为较差但真实常见的错误回答，作为 rejected

要求：
1. chosen 必须更准确、更安全、更切题。
2. rejected 必须像真实模型可能犯的错误，不要故意写得过于愚蠢。
3. rejected 的问题可以包括：
   - 过度自信
   - 缺少危险信号提醒
   - 建议过于笼统
   - 没有追问意识
4. 不要让 rejected 出现违法、极端、无意义内容。
5. 输出为 JSON：
   {
     "system": "...",
     "history": [],
     "question": "...",
     "response_chosen": "...",
     "response_rejected": "..."
   }

问题：
{{question}}
```

### 6.5 偏好对审核 Prompt 模板

```text
你现在是偏好数据质检员。

请检查下面每条偏好数据是否满足：
1. chosen 确实优于 rejected
2. 两个回答是针对同一个问题
3. chosen 和 rejected 的差异是质量差异，不是任务差异
4. 没有明显医学错误被放进 chosen
5. rejected 不是过于夸张的低质量答案

输出格式：
- id
- verdict: PASS / FAIL / REVISE
- issues
- corrected_pair

待审核数据：
{{candidate_pairs}}
```

## 7. 数据审核与去重模板

### 7.1 为什么要单独做审核

只生成不审核，最后训练出来的模型会学到：

- 模板化回答
- 医学幻觉
- 风险边界缺失
- 重复表达
- 假多样性

### 7.2 通用审核 Prompt 模板

```text
你现在是训练数据质量审核员。

请审核下面的数据，重点检查：
1. 格式是否合法
2. 是否重复或高度相似
3. 是否含明显事实错误
4. 是否包含不安全建议
5. 是否过度模板化
6. 是否适合作为训练样本

输出要求：
对每条输出：
- id
- quality_score: 1-5
- verdict: KEEP / DROP / REWRITE
- reason

数据：
{{samples}}
```

### 7.3 去模板化改写 Prompt 模板

当你发现大量回答都像一个模子里出来的，用这个模板：

```text
你现在是训练数据改写助手。

请在不改变医学含义和安全边界的前提下，对下面回答进行自然改写。

要求：
1. 保留核心医学信息。
2. 保留风险提醒和就医边界。
3. 改变表达方式，避免模板化重复。
4. 不要新增事实。
5. 输出只给出改写后的回答。

原回答：
{{answer}}
```

## 8. Eval 数据生成模板

### 8.1 Eval 数据的用途

评测集不是训练集。  
它应该用来测：

- base 和 tuned model 的真实差异
- 高风险问题上的表现
- 追问能力
- 安全边界

### 8.2 Eval Prompt 扩写模板

```text
你现在是医学评测集设计助手。

请围绕给定主题，生成 20 条互不重复的评测问题。

要求：
1. 问题必须自然，像真实用户表达。
2. 不要生成标准答案。
3. 要覆盖不同表达方式、年龄场景和风险等级。
4. 不要与示例问题过于相似。
5. 输出 JSON 数组，每条包含：
   - id
   - category
   - risk_level
   - prompt

主题：
{{topic}}

示例问题：
{{examples}}
```

## 9. 生产高质量数据的几个硬规则

### 9.1 先少量高质量，再大量扩写

先用人工写 50 到 200 条高质量种子，再让 LLM 扩写。  
不要从零开始让 LLM 自由生成 1 万条。

### 9.2 训练集和评测集必须隔离

LLM 很容易把表达方式重复到 eval 里，所以你要：

- 先定 eval 主题
- 单独生成 eval
- 不让 eval 回流训练

### 9.3 对高风险问题必须强审

这些主题必须人工强审：

- 儿科高热
- 孕产妇异常情况
- 胸痛、呼吸困难、抽搐、意识障碍
- 用药与剂量
- 严重感染风险

### 9.4 拒绝“假偏好”

偏好数据里不要全是：

- 一个正常回答
- 一个特别离谱的回答

这种数据太假，学不到真实区分能力。

### 9.5 保留 bad cases 库

每轮训练后，把模型失败案例收集起来，再让 LLM 围绕这些失败点生成补充数据。  
这比盲目扩量更有效。

## 10. 推荐的最小执行方案

如果你现在就要开工，建议按这个顺序：

1. 人工写 100 条种子 SFT 样本
2. 用 SFT 扩写 Prompt 扩成 1000 条候选
3. 用 SFT 审核 Prompt 筛选和修订
4. 人工抽检 10% 到 20%
5. 训练第一版 SFT
6. 收集 200 条 bad cases
7. 用 `base` 和 `SFT` 对同一批 prompt 生成候选回答
8. 用偏好判断 Prompt 构造 preference 数据
9. 审核 preference 数据
10. 跑 DPO

## 11. 最后提醒

优质数据的本质不是“像人写的”，而是：

- 目标明确
- 质量可控
- 风格稳定
- 风险边界清楚
- 能真实改善你要优化的能力

LLM 的最大价值不是替你免掉数据工作，而是把“高质量数据生产”从手工小作坊，变成一个能反复迭代的流水线。
